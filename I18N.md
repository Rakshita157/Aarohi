# Internationalization (i18n) вҖ” Implementation Guide

## Overview

Aarohi supports **English (en)** and **Hindi (hi)** across all frontend UI and the AI Sakhi chatbot backend. The implementation uses `react-i18next` on the frontend and a lightweight translation layer on the Express backend.

**Initialization flow:**
1. `main.jsx` imports `./i18n` вҖ” this triggers `i18n.js` to run
2. `LanguageDetector` checks localStorage for `aarohi_lang`, falls back to browser language
3. `initReactI18next` hooks i18next into React's context system
4. Every component that calls `useTranslation()` gets the current `t()` function reactively

**Rendering flow:**
1. Component calls `const { t } = useTranslation()`
2. `t('nav.home')` looks up `translation.en.nav.home` вҶ’ `"Home"`
3. When language changes, all components using `t()` re-render automatically

## How Language Switching Works

```
User clicks language dropdown
        в”Ӯ
        в–ј
Navbar.jsx: i18n.changeLanguage('hi')
        в”Ӯ
        в”ңв”Җв–ә i18next updates internal state
        в”ңв”Җв–ә localStorage.setItem('aarohi_lang', 'hi')
        в”ңв”Җв–ә Fires 'languageChanged' event
        в”Ӯ     в””в”Җв–ә i18n.js listener sets <html lang="hi">
        в”ңв”Җв–ә All useTranslation() hooks re-render
        в”Ӯ     в””в”Җв–ә t() now reads from hi/translation.json
        в””в”Җв–ә Next API call includes Accept-Language: hi header
              в””в”Җв–ә Backend builds Hindi system prompt for Sakhi
```

**Key points:**
- Language change is **instant** вҖ” React re-renders all `t()` consumers in the same tick
- No page reload required
- The `<select>` in Navbar is a controlled component bound to `i18n.language`

## How Translations Are Organized

Translations use a **flat namespace per page** structure:

```
client/src/locales/
в”ңв”Җв”Җ en/
в”Ӯ   в””в”Җв”Җ translation.json      вҶҗ English (source of truth)
в””в”Җв”Җ hi/
    в””в”Җв”Җ translation.json      вҶҗ Hindi (same keys, translated values)
```

**Namespace structure inside each JSON file:**

```json
{
  "nav":        { ... },   //  9 keys  вҖ” Navbar links, buttons
  "footer":     { ... },  // 20 keys  вҖ” Footer columns, links
  "home":       { ... },  // 53 keys  вҖ” Home page hero, sections
  "about":      { ... },  // 64 keys  вҖ” About page
  "learn":      { ... },  // 37 keys  вҖ” Learn page grid, categories
  "learnSidebar": { ... },// 30 keys  вҖ” Sidebar progress, path, features
  "lesson":     { ... },  // 33 keys  вҖ” Lesson page modules, glossary
  "resources":  { ... },  // 37 keys  вҖ” Resources page myths, filters
  "askSakhi":   { ... },  // 17 keys  вҖ” Ask Sakhi chat page
  "chatBox":    { ... },  // 11 keys  вҖ” Chat widget (floating)
  "login":      { ... },  // 11 keys  вҖ” Login page
  "signup":     { ... },  // 16 keys  вҖ” Signup page
  "community":  { ... },  //  3 keys  вҖ” Community page
  "notFound":   { ... },  //  4 keys  вҖ” 404 page
}
```

**Total: 345 keys across 14 namespaces.**

**Naming convention:**
- Top-level keys match the page/component: `nav`, `footer`, `home`, `about`, etc.
- Sub-keys use camelCase: `heroTitle`, `heroSubtitle`, `startLearning`
- Interpolation params use `{{param}}`: `"copyright": "В© {{year}} Aarohi"`

## How `useTranslation()` Works

```jsx
import { useTranslation } from 'react-i18next';

const MyComponent = () => {
  const { t, i18n } = useTranslation();

  return (
    <div>
      <h1>{t('home.heroTitle')}</h1>           {/* Simple key lookup */}
      <p>{t('footer.copyright', { year: 2025 })}</p> {/* With interpolation */}
      <button onClick={() => i18n.changeLanguage('hi')}>
        Hindi
      </button>
    </div>
  );
};
```

**What `t()` does internally:**
1. Resolves the key path: `'home.heroTitle'` вҶ’ `resources[lang].translation.home.heroTitle`
2. Replaces interpolation tokens: `{{year}}` вҶ’ `2025`
3. If key is missing, falls back to `fallbackLng` ("en") and logs a console warning
4. Returns the translated string

**What `i18n` object provides:**
| Property | Purpose |
|----------|---------|
| `i18n.language` | Current language code (`"en"` or `"hi"`) |
| `i18n.changeLanguage(lng)` | Switch language, update localStorage, re-render |
| `i18n.isInitialized` | Whether i18next has finished initializing |

## How Language Persistence Works

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ              Persistence Flow                     в”Ӯ
в”Ӯ                                                  в”Ӯ
в”Ӯ  First visit:                                    в”Ӯ
в”Ӯ    localStorage('aarohi_lang') = null            в”Ӯ
в”Ӯ    вҶ’ LanguageDetector checks navigator.language  в”Ӯ
в”Ӯ    вҶ’ Defaults to "en"                            в”Ӯ
в”Ӯ                                                  в”Ӯ
в”Ӯ  User switches to Hindi:                         в”Ӯ
в”Ӯ    i18n.changeLanguage('hi')                     в”Ӯ
в”Ӯ    вҶ’ localStorage.setItem('aarohi_lang', 'hi')   в”Ӯ
в”Ӯ                                                  в”Ӯ
в”Ӯ  Page refresh:                                   в”Ӯ
в”Ӯ    LanguageDetector reads localStorage('aarohi_lang') в”Ӯ
в”Ӯ    вҶ’ Finds "hi" вҶ’ loads Hindi translations        в”Ӯ
в”Ӯ                                                  в”Ӯ
в”Ӯ  Tab/window close & reopen:                      в”Ӯ
в”Ӯ    Same as refresh вҖ” localStorage persists       в”Ӯ
в”Ӯ                                                  в”Ӯ
в”Ӯ  Clear localStorage:                             в”Ӯ
в”Ӯ    localStorage('aarohi_lang') = null            в”Ӯ
в”Ӯ    вҶ’ Falls back to navigator.language вҶ’ "en"     в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

**Detection order** (configured in `i18n.js`):
1. `localStorage` key `aarohi_lang` вҖ” highest priority
2. `navigator.language` вҖ” browser's default language

**Backend persistence:**
- The Axios interceptor in `api.js` reads `aarohi_lang` from localStorage on every request
- Sends it as `Accept-Language` header
- Backend stores language in the UserMemory model's `preferences.language` field (future use)

## Advantages

- **Zero page reloads** вҖ” language switches instantly via React re-rendering
- **Centralized translations** вҖ” all text in JSON files, easy for translators to work with
- **Type-safe keys** вҖ” missing keys log console warnings during development
- **Fallback chain** вҖ” missing Hindi keys fall back to English automatically
- **Backend-aware** вҖ” AI Sakhi responds in the selected language
- **Non-invasive** вҖ” no API breakage, existing functionality unchanged
- **Scalable** вҖ” adding a new language requires only a new JSON file + config update
- **Standards-based** вҖ” uses `Accept-Language` header (HTTP standard) for backend communication

## Limitations

- **Hindi translations are empty** вҖ” all 345 Hindi values are placeholder strings (`""`). They need to be filled in by a translator.
- **Lesson/resource content is not translated** вҖ” `lessons.js` and `resources.js` contain English-only content data. These are not UI chrome but educational content that would need separate translation effort.
- **Decorative SVG text** вҖ” The phone mockup illustration on the Learn page contains hardcoded English text. This is a design element, not functional UI.
- **Backend error messages** вҖ” API error responses ("Conversation not found", "Invalid email or password") remain in English. The frontend displays these in a toast/alert and could be translated client-side in the future.
- **No pluralization rules** вҖ” Hindi and English have different plural forms. Current implementation uses simple string substitution.
- **No RTL support** вҖ” Hindi is LTR like English. Adding Arabic/Urdu would require RTL layout changes.
- **No date/number formatting** вҖ” Dates and numbers are not locale-formatted (e.g., Hindi numerals).

## Adding a New Language

To add a new language (e.g., **Bengali вҖ” `bn`**):

### Step 1: Create the translation file

```bash
# Copy the English file as a starting point
cp client/src/locales/en/translation.json client/src/locales/bn/translation.json
```

Then translate all 345 values in the new file.

### Step 2: Register the language in i18n.js

```js
// client/src/i18n.js
import bn from "./locales/bn/translation.json";

const resources = {
  en: { translation: en },
  hi: { translation: hi },
  bn: { translation: bn },   // вҶҗ add this
};

i18n.init({
  // ...
  supportedLngs: ["en", "hi", "bn"],  // вҶҗ add this
});
```

### Step 3: Add backend chatbot translations

```js
// server/services/chatTranslations.js
const translations = {
  en: { ... },
  hi: { ... },
  bn: {
    disclaimer: "...",
    outOfScope: "...",
    crisis: "...",
    langInstruction: "...",
    systemLangNote: "...",
  },
};
```

### Step 4: Add the language option in the UI

```jsx
// client/src/components/Navbar.jsx вҖ” inside the <select>
<option value="bn">аҰ¬аҰҫаҰӮаҰІаҰҫ</option>
```

**That's it.** No other files need changes вҖ” the `Accept-Language` header, backend middleware, and Gemini prompt all work dynamically with whatever language code is sent.

## Best Practices

1. **Always use `t()` for user-visible text** вҖ” no exceptions for buttons, labels, headings, errors, or alt text
2. **Use descriptive key names** вҖ” `home.heroTitle` not `home.text1`
3. **Keep keys flat per namespace** вҖ” avoid more than 2 levels of nesting
4. **Never hardcode fallback text in components** вҖ” let i18next handle fallback to English
5. **Use interpolation for dynamic values** вҖ” `t('greeting', { name })` not template literals
6. **Test both languages** вҖ” switch to Hindi and visit every page to catch missing keys
7. **Keep en/hi structures identical** вҖ” same keys at the same paths in both files
8. **Language names in native script** вҖ” "English" / "аӨ№аӨҝаӨЁаҘҚаӨҰаҘҖ" not "English" / "Hindi" (accessibility standard)
9. **Don't translate technical strings** вҖ” URLs, CSS classes, variable names, email addresses
10. **Backend translations in a separate file** вҖ” keep `chatTranslations.js` isolated from frontend locale files
